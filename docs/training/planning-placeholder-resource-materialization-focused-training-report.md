# Planning Placeholder Resource Materialization Focused Training Report

## Incident

The Query Data Runtime Baseline generated by plugin 0.9.51 correctly installed and executed its Query Data workflow, but the App Plan's `Not applicable.` Dashboard decision was emitted as a real Type 103 page. Yeeflow consequently displayed `Not applicable.` as an application navigation surface.

## Root cause

The materializer already had a partial `isNonResourceName()` guard, but it required exact text such as `Not applicable`. The period in `Not applicable.` bypassed the regular expression. The resulting false resource entered Dashboard counts, API-issued page ID allocation, `Pages[]`, and runtime page discovery.

The pre-signing suite also lacked an independent output-side resource identity gate, so the generated package passed despite the visible placeholder resource.

## Focused fix

- Added shared `planning-placeholder-utils.mjs` normalization.
- Applied it to full-app materialization, Approval layout parsing, and generated-final completeness parsing.
- Added `validate-planning-placeholder-materialization.mjs`.
- Added the new gate to first-generation YAPK preflight.
- Added regressions for Pages, FormNewReports, navigation, punctuation variants, no-resource Dashboard generation, and legitimate similar business names.

## Proof

- The original 0.9.51 Query Data package now fails with `PLANNING_PLACEHOLDER_MATERIALIZED_AS_RESOURCE` at `$.Pages[0].Title`.
- A fresh App Plan fixture containing `### 14.1 Not applicable.` materializes with `Pages: []` and no placeholder navigation item.
- `Not Applicable Cases` remains a valid resource title.

## Boundary

This training changes planning-placeholder interpretation and generated-final validation only. It does not alter the Query Data node serialization or the already proven Query Data runtime behavior.
