# App Builder Core Extraction Readiness Checklist

## Contract and boundary evidence

- [x] Core accepts only canonical domain intent.
- [x] Execution context, capability profile, authority, origin, and model invocation live outside every Core package.
- [x] The package graph enforces one-way adapter to service to Core dependencies.
- [x] No closed host or channel enum exists.
- [x] Model/provider policy remains host-owned and opaque to Core.
- [x] Unsupported capability, protocol mismatch, unknown fields, and authority escalation fail closed.
- [x] Core input/output isolation is covered by a capturing test double.

## Implementation evidence

- [x] An in-process execution kernel is runnable.
- [x] A host-controlled adapter is runnable with an injected deterministic model port.
- [x] A managed-provider fake adapter is runnable without live integration.
- [x] A future-host fixture is runnable without a Core change.
- [x] Identical structured intent produces identical plan, validation, materialization, verification, and repair output across adapters.
- [x] Official execution artifacts are generated by a builder and validated by checksums and exact exports.
- [x] Cross-host equivalence is covered in compiled source; the Plugin-owned Codex path has official distribution, archive, and simulated-installed parity.

## Repository-extraction decision gates

- [ ] Establish an independently versioned package and release policy for the future Core repository.
- [ ] Produce a reproducible clean-checkout build without relying on ignored historical workspace scaffolding.
- [ ] Define ownership and migration treatment for the legacy `app-builder-core-runtime-client` package name while keeping concrete communication outside Core.
- [ ] Decide whether the execution service moves with Core or remains in a separate runtime repository, without creating a dependency cycle.
- [ ] Prove the extracted repository consumes the same sanitized fixtures and reproduces official artifact checksums.
- [ ] Define downstream compatibility and rollback policy for the monorepo consumer.

Live provider parity, Web deployment, tenant mutation, signing, installation, and production runtime proof are deliberately not extraction prerequisites for this incubation and are not claimed.

## Verdict

**NOT READY for standalone repository extraction.** The complete monorepo incubation gate run passed, but all six unchecked repository-extraction decision gates above remain open. The in-process service boundary is verified; passing that boundary does not authorize repository creation.
